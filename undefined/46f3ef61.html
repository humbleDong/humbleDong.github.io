SpringBootWeb核心应用

概述

SpringBoot支持的两种Web类型：

  • SERVLET(即传统的WebMVC)同步阻塞I/O模型
  • REACTIVE(响应式的WebFlux)异步非阻塞I/O模型

性能上REACTIVE优于SERVLET，但是WebFlux不是很完善，也不是很成熟

SpringBoot通过spring-boot-start-web对SpringMVC的配置内容大致为：

  • 包含对ContentNegotiatingViewResolve和BeanNameView的Bean注册
  • 支持对静态资源，包括对静态网页index.html的支持，以及对WebJars的支持
  • 包含对Converter、GenericConverter、Formatter的Bean自动注册
  • 支持对HTTPMessageConverters消息转换器
  • 支持对MessageCodesResolver的自动注册
  • 自动使用ConfigurationWebBindingInitializer的Bean

​ 既然SpringBoot只提供了对SpringMVC的自动配置，所以我们尝试在SpringBoot中使用SpringMVC和独立使用是没有什么区别的，先导入web的启动器，然后定义两个接口，分别返回JSON、XML格式的数据

注解还是MVC那一套，Springboot只是完成了框架上的配置而已

概述

Springboot支持的容器：

容器 依赖

容器配置

容器参数绑定类为ServletProperties类，可以通过server.*开头的参数来配置所有容器的通用配置

更多的参看ServletProperties类，Tomcat容器的可以通过：server.tomcat.*参数配置

也可以通过Java类自定义Servlet容器，只需要实现WebServerFactoryCustomizer接口即可

除了ConfigurableWebServerFactory，还有TomcatServletWebServerFactory，JettyServletWebServerFactory和UndertowServletWebServerFactory。

切换容器

直接在spring-boot-starter-web启动依赖包中把Tomcat依赖替换成其他容器即可

随机空闲端口

为了防止端口冲突，Springboot可以配置并使用系统随机空闲的端口，只需要把server.port设置为0即可

SpringBoot会自动扫描系统空闲的端口，并且随机选择一个使用

SSL

通过server.ssl.*参数来配置SSL支持HTTPS的访问

本地环境中可以用JDK的keytool工具生成一个自签名的证书

然后把证书复制到应用资源根目录

再配置application即可，这样你的应用就是支持HTTPS不是原来的HTTP了

当然也可以支持同时支持

SpringBoot建议，把HTTPS用配置文件形式，HTTP用java程序配置，然后再启动应用就可以看到同时支持两个协议的日志

一般情况Springboot应用不会直接对外开放，一般是内网部署，都需要负载均衡组件或者服务网关进行路由转发，所以HTTPS会配置在外网负载均衡的组件上

分布式会话

传统的会话机制保存在应用服务器端，会话共享需要借助Servlet容器来实现，比如Tomcat提供了Session复制的功能，简单的集群复制没有什么问题，但是，如果集群节点过多，就会因网络延迟造成会话不能及时同步，进而影响用户体验。

Spring Session让集群变得轻而易举，它是Spring提供的一个用于管理用户会话的框架，可以将Session数据存储到第三方存储中替换容器中的HttpSession机制，而无需绑定特定对的应用容器。

SpringBoot支持的Session存储：

  • Redis
  • JDBC
  • Hazelcast
  • MongoDB

如果类路径中存在以上单个SpringSession模块，则SpringBoot会自动使用该存储实现，如果有多个实现，则SpringBoot按以上列举顺序的组件来选择，如果一个没有就选择默认的SpringSession

Session的自动配置类为org.springframework.boot.autoconfigure.session.SessionAutoConfiguration,它同样被注册在新的org.springframework.boot.autoconfigure.AutoConfiguration.imports自动配置文件中，SpringBoot自动配置后，就不需要使用@Enable*HttpSession注解了,redis也是这样

Session参数绑定类为SessionProperties，通过一系列spring.session.*参数配置Session会话，每个存储也都有自己的扩展实现，比如Redis就是spring.session.redis.*

在此之前一定要有Redis容器，自行安装。

跨域

首先了解同源策略

同源策略（Same-Origin Policy）是一种在Web浏览器中实施的安全机制，它旨在防止一个网页上的脚本通过XMLHttpRequest或Fetch API等方式去请求另一个域下的资源。同源策略有助于防止恶意网站利用用户身份在其他站点上执行恶意操作。

同源策略的要求是：

  1. 协议相同：两个页面的协议必须相同，例如，一个页面使用HTTP协议，另一个页面也必须使用HTTP协议，而不是HTTPS协议。
  2. 域名相同：两个页面的域名必须相同，即二级域名、三级域名等必须相同。
  3. 端口相同：如果两个页面使用了端口号，那么端口号也必须相同。

如果以上任一条件不满足，就认为两个页面不同源，同源策略将会阻止它们之间的一些操作。这有助于减少潜在的安全风险，比如阻止恶意网站通过脚本访问用户在其他站点上的敏感信息。

同源策略并不是绝对的安全措施，因为有时候确实需要不同源的页面之间进行一些交互。为了在不同源的页面之间实现交互，可以使用一些跨域通信的技术，如JSONP、CORS（跨域资源共享）等。这些技术允许开发者有限度地突破同源策略的限制，以实现安全的跨域通信。

SpringMVC本身是支持跨域请求的，只需要在需要跨域的方法上使用@CrossOrigin注解即可

默认的@CrossOrigin支持：

  • 所有的请求来源
  • 所有的HTTP头
  • 所有的HTTP方法

如果有需要可以在@CrossOrigin注解中指定跨域的详细参数，除了作用在方法上，也可以作用在类上，这样它的所有方法都可以跨域了，也可以在WebMvcConfigurer配置类中全局添加跨域注册的功能

但是一般接口不会对外暴露，这时候统一用Nginx上配置跨域
